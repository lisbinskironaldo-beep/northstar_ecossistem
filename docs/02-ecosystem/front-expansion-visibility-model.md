# Front Expansion Visibility Model

## Purpose

This document defines how Northstar should signal future fronts inside the product without cluttering the current experience.

The user should feel:

- this ecosystem is growing
- more is coming
- Echo is only the first living front

The user should not feel:

- the product is unfinished chaos
- the current app is trying to do everything already

## Core Rule

Future fronts should be visible as promise, not as noise.

## Current Practical Model

Inside Echo, future fronts should appear as:

- subtle horizon cards
- locked or dimmed surfaces
- creator-facing roadmap hints

Not as:

- dead navigation tabs
- broken empty pages
- giant "coming soon" banners everywhere

## Listener Rule

For listeners:

- keep the current experience focused on music
- show only a light sense that the world is bigger than Echo

Good examples:

- small horizon strip
- subtle locked symbols
- visual suggestion that the artist journey can grow further

Current Echo mobile implementation:

- a small horizon strip in listener surfaces
- no extra dead tabs
- no giant coming-soon walls

## Creator Rule

For creators:

- make the future more explicit
- show that Echo is the starting point
- show that the same creator workspace will later connect to Pulse and Lumen

This helps the creator feel:

- I am not building only inside one app
- I am building inside a larger creator system

Current Echo mobile implementation:

- the creator mode now shows Echo as active and Pulse/Lumen as horizon expansions

## Product Rule

Future visibility must:

- increase ambition
- not increase confusion

If a future-front signal makes the current app harder to understand, it is too loud.

## Reuse Rule Across Fronts

The current Echo mobile work already reveals some patterns that should later be reused across fronts.

Reusable patterns:

- `following` versus `discover` split
- creator or artist home as one coherent stage
- lightweight event visibility in the main feed
- subtle horizon strip for future-front visibility
- fixed lightweight playback or session dock

Echo-only patterns for now:

- 20-second music preview behavior
- room-based access labels like `kids`, `parody`, `clone-inspired` and `explicit`
- borrowed album language
- music-first artist-home launch list

This matters because:

- Pulse can later reuse discovery rhythm and event visibility
- Lumen can later reuse creator-home structure and event surfaces
- neither one should be forced to look like a music app

So the rule stays:

- reuse the pattern
- not the exact Echo surface
