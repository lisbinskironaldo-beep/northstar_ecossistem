# Echo Creator Onboarding State Model

## Purpose

This file defines the exact meaning of each creator state in the first controlled Echo beta.

It exists so the operator does not confuse interest with readiness.

## Core Rule

Do not collapse these states into one idea:

- invited
- accepted
- onboarded

Each one means something different.

## State Meanings

### Candidate

The creator is only being considered.

Nothing real has happened yet.

### Invited

The first real message was sent.

This means outreach happened, not trust or readiness.

### Accepted

The creator replied positively or agreed to try Echo.

This still does not mean the creator is ready for beta operations.

### Onboarded

The creator has crossed the actual intake bar:

- active persona selected
- basic bio present
- at least one active main-room track
- at least two tracks in the workspace
- active versus sleeping catalog logic understood

### Inactive

The creator stopped moving or is not worth continuing for this wave.

### Hold

The creator is still potentially valuable, but not for the current wave.

## Operator Move Rules

Move from:

- `candidate` -> `invited`
  only after the first real outreach message is sent

- `invited` -> `accepted`
  only after a clear positive reply

- `accepted` -> `onboarded`
  only after the intake bar is materially complete

## Week 01 Link

Once a creator becomes `onboarded`, the operator should start reading:

- week-one pulse
- creator goal
- operator watchlist

This is the first practical sign that the creator is now inside the product, not only inside the pipeline.

## Product Support

The current Echo creator workspace now exposes:

- onboarding state label
- onboarding path steps
- week-one pulse
- creator goal
- operator watchlist

## Final Rule

The first beta should optimize for:

- clear progress
- low rescue
- honest product learning

not for inflated creator counts.
